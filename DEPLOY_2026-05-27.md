## Deploy (orden estricto — DB primero, código después)

```bash
# 1. Subir commits al remote
cd ~/Documentos/GitHub/arabitofacturacion
git push origin master

# 2. En el servidor de producción
cd /var/www/arabitofacturacion

# 3. Pull del nuevo código
git fetch origin
git log HEAD..origin/master --oneline   # ← revisar qué entra

# 4. MIGRAR PRIMERO (antes de pull) — si el código nuevo asume la columna y aún no existe, error 500
php artisan migrate
# Confirmar:
#   Migrating: 2026_05_27_120000_add_composite_index_pedidos_fast
#   Migrated:  2026_05_27_120000_add_composite_index_pedidos_fast
#   Migrating: 2026_05_27_140000_add_imborrable_to_pago_pedidos_table
#   Migrated:  2026_05_27_140000_add_imborrable_to_pago_pedidos_table

# 5. Pull del código
git pull origin master

# 6. Composer si hubo cambios en autoload (poco probable, pero por las dudas)
composer install --no-dev --optimize-autoloader

# 7. Cache reset (importante — getPedidoFun memoiza con Cache::get/put)
php artisan cache:clear
php artisan config:cache
php artisan route:cache
php artisan view:clear

# 8. Restart PHP-FPM (para que fastcgi_finish_request se aplique con el nuevo código)
sudo systemctl restart php8.x-fpm
# o el comando que use el sucursal
```

## Rollback (si algo explota)

### Opción A: Rollback solo del código (DB ya migrada — generalmente OK)

```bash
cd /var/www/arabitofacturacion
git reset --hard $(cat /tmp/prod_sha_before.txt)
php artisan cache:clear
php artisan config:cache
sudo systemctl restart php8.x-fpm
```

La columna `imborrable` y el índice nuevo no rompen el código viejo (el código viejo simplemente los ignora). Quedan en la BD inocuos. **Esta es la ruta más rápida y segura.**

### Opción B: Rollback completo (código + DB)

Solo si la nueva columna o índice causa problemas raros (improbable):

```bash
cd /var/www/arabitofacturacion
git reset --hard $(cat /tmp/prod_sha_before.txt)
php artisan migrate:rollback --step=2   # ← reverse de las 2 migraciones de hoy
php artisan cache:clear
php artisan config:cache
sudo systemctl restart php8.x-fpm
```

## Soft issues conocidos (no bloqueantes — log para seguimiento)

| # | Issue | Severidad | Mitigación futura |

| 1 | `[setPagoPedido] Error en la petición (catch): undefined e is not a function` aparece en consola del browser tras setPagoPedido exitoso en algunos flujos. ## Validaciones de éxito (post-deploy)

Ejecutar 24h después de deploy:

```bash
# 1. Verificar que ningún pago POS aprobado se "evaporó" por bug
mysql -u root -p arabitofacturacion -e "
  SELECT COUNT(*) AS pos_imborrables_hoy
  FROM pago_pedidos
  WHERE imborrable = 1
    AND created_at >= CURDATE();
"
# Esperado: número > 0 si hubo ventas POS hoy

# 2. Verificar que ningún pedido quedó huérfano sin items y sin is_devolucion_pura
mysql -u root -p arabitofacturacion -e "
  SELECT p.id, p.estado, p.created_at, COUNT(i.id) AS items
  FROM pedidos p
  LEFT JOIN items_pedidos i ON i.id_pedido = p.id
  WHERE p.estado = 1
    AND p.created_at >= CURDATE()
  GROUP BY p.id
  HAVING items = 0
  LIMIT 20;
"
# Esperado: solo refunds puros legítimos (clean_total negativo)
```
